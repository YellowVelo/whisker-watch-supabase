Spec: Build a real user profile (name, timezone, settings UI)
Problem
profiles currently has only id, email, account_type, role, first_name (added in migration 0015, silently populated from signup metadata, never user-editable). There is no:

Full name (no last_name)
Stored timezone or location
Any settings UI where a user can view or edit their own profile
This causes two concrete gaps found in this session:

Notifications/emails fall back to raw email when first_name is unset (now partially mitigated — see "Already fixed" below — but still degrades to email for anyone who never had a first_name, e.g. signed up before 0015 or via a flow that didn't pass it).
"Today" / day-boundary logic (separate bug, already speced/fixed elsewhere) has no durable, user-controlled timezone to fall back on outside of an active browser session — e.g. a future server-side job (scheduled notification, day-rollover) has nothing to read.
Already fixed, do not duplicate
first_name is now used (with email fallback) in co-owner/account-deletion notifications and the invite-email template — see supabase/functions/{delete-pet,delete-account,invite-co-owner}/index.ts. This spec is about making that name reliably present and editable, not about wiring it into more call sites (do that opportunistically if you find more raw-email usages, but it's not the core scope here).

Schema changes
alter table public.profiles
  add column last_name text check (char_length(last_name) <= 100),
  add column timezone text; -- IANA tz name, e.g. "America/New_York"
No RLS change needed — profiles_update_own (migration 0001) already covers self-editable, non-privileged columns, same reasoning as 0015's comment for first_name.
Keep first_name/last_name nullable — do not force a migration backfill; unset stays a valid, handled state (falls back to email, per already-fixed code).
Decide during implementation: single display_name field instead of first_name/last_name split? Recommend keeping the split (matches existing first_name usage in the Home greeting, which should stay first-name-only) but add a derived helper (fullName = [first_name, last_name].filter(Boolean).join(' ')) for contexts that want the full name.
Timezone capture
Auto-detect via Intl.DateTimeFormat().resolvedOptions().timeZone in AuthContext.jsx's loadUserWithProfile (same choke point already used for claim_pending_co_owner_invites()), and upsert it to profiles.timezone if:

the stored value is null (first time), or
it differs from the detected value (keeps it fresh as the user travels, without needing a manual re-detect button).
This is a plain client-side update, not a new RPC — profiles_update_own already permits it.

Settings UI (new)
There's currently zero profile-editing UI (checked src/pages/Settings.jsx, Account.jsx, Preferences.jsx — none expose first_name/account-level identity fields today). Add a "Profile" section, likely on Account.jsx or a new sub-page linked from Settings.jsx, with:

First name, last name — plain text inputs, save via existing profiles_update_own policy.
Timezone — read-only display of the auto-detected value plus a manual override dropdown (searchable list of IANA zones) for the travel/VPN case. Manual override, once set, should stop being silently overwritten by the auto-detect logic above (add a timezone_is_manual boolean, or simpler: only auto-update if the user hasn't touched the field — track via a timezone_set_at timestamp vs. a "last auto-detected at" comparison; pick whichever is simpler to implement correctly, note the tradeoff either way).
No password/email editing in this pass — that's Supabase Auth's own concern (already has dedicated flows per ResetPassword.jsx/ForgotPassword.jsx); don't duplicate it here.
Acceptance criteria
A brand-new signup ends up with first_name set (existing behavior via signup metadata) and timezone auto-populated on first login, with no user action required.
An existing user (pre-0016, no timezone set) gets timezone populated on their next login without needing to do anything.
A user can view and edit first/last name and timezone from Settings, and the change persists (reload confirms it, another device's session picks it up).
Setting a manual timezone override survives subsequent logins (isn't silently reset by auto-detection).
Existing notification/invite-email code paths (already fixed this session) automatically benefit from more users having a real name, with no further code changes required there.
Out of scope
Location/address beyond timezone (no evidence any current or planned feature needs street address, etc. — don't build it speculatively).
Avatar/photo upload for the user themselves (pet photos already exist via storageClient.js; a user avatar is a separate, unrequested feature).
Any server-side scheduled job that actually consumes profiles.timezone (e.g., "send check-in reminder at 8 AM their time") — that's a future feature this profile groundwork would unblock, not part of this spec.
