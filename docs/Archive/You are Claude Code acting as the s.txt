# 5. Baseline and Daily Score Model

## 5.1 Baseline Requirement

Every pet’s established baseline defines what is normal for that individual pet.

The system must not use one universal definition of normal for all pets.

For each scored day:

* The daily check-in begins with the pet’s baseline-normal state.
* Unchanged attributes inherit the pet’s baseline.
* The owner records only attributes that changed.
* Only recorded negative Health observations deduct points.

The baseline is the behavioral starting assumption.

The numerical score starts at 10.

## 5.2 Check-In Required

A Health Score may only exist for a date with a completed Daily Check-In.

No check-in means unknown.

No check-in must not be interpreted as:

* Normal
* 10/10
* Equal to yesterday

## 5.3 Starting Score

Every completed, scored day starts at:

**10**

Displayed format:

**{score}/10**

Examples:

* 10/10
* 9/10
* 7/10
* 4/10
* 0/10

Do not display the score as:

* 100
* 90
* 70
* A percentage

# 6. Health Score Calculation

## 6.1 Calculation Formula

```text
Health Score = 10 - total Health Attribute deductions
```

Clamp the result between 0 and 10.

```text
minimum = 0
maximum = 10
```

Example:

```text
Starting score:       10
Appetite deduction:    1
Vomiting deduction:    2
Total deductions:      3
Health Score:          7/10
```

## 6.2 Per-Attribute Maximum

Each Health Attribute can deduct a maximum of:

**2 points per pet per date**

This maximum applies even when:

* Multiple observations exist for the same attribute.
* The user edits the check-in.
* Duplicate records are accidentally submitted.
* Multiple option values are associated with one attribute.

## 6.3 Included Attributes

Only the configured Health Attributes may deduct points:

```typescript
[
  'appetite',
  'water_intake',
  'bathroom',
  'stool',
  'vomiting'
]
```

Ignore score deductions attached to:

* Energy
* Mobility
* Breathing
* Itching
* Weight
* Behavior
* Medication exceptions
* Other
* Unknown observation types

Log a structured warning when an unknown or excluded attribute contains a Health Score deduction.

## 6.4 Configured Deduction Values

Deduction values must be explicitly stored against observation options.

Do not infer deductions at runtime solely from option order.

Supported values:

```text
0
1
2
```

Meaning:

* Baseline/normal option: 0
* Mild negative observation: 1
* More significant negative observation: 2

When four negative choices exist, the default product mapping is:

| Negative choice | Deduction |
| --------------- | --------: |
| First           |         1 |
| Second          |         1 |
| Third           |         2 |
| Fourth          |         2 |

The explicit configured value remains the source of truth.

# 8. Directional Comparison Rules

## 8.1 Supported Direction States

```typescript
'up' | 'equal' | 'down' | 'unknown'
```

## 8.2 User-Facing Indicators

| State   | Icon             | Color | Text             |
| ------- | ---------------- | ----- | ---------------- |
| Up      | Up arrow         | Green | versus yesterday |
| Equal   | Equal sign       | Blue  | versus yesterday |
| Down    | Down arrow       | Red   | versus yesterday |
| Unknown | Dash or no arrow | Gray  | Not enough data  |

The icon and accessible label must communicate direction in addition to color.

## 8.3 Direction Is Not Medical Interpretation

The direction describes movement only.

Examples:

* Appetite up means the recorded appetite level increased.
* Vomiting up means vomiting frequency increased.
* Water up means recorded water intake increased.
* Mobility down means recorded mobility decreased.
* Health Score up means the score is numerically higher.

Do not translate directional states into:

* Better
* Worse
* Improving
* Declining
* Healthy
* Concerning
* Stable

## 8.4 Health Score Direction

Compare today’s Health Score with the most recent immediately preceding calendar day only.

```typescript
computeHealthScoreDirection(
  todayScore,
  yesterdayScore,
  todayStatus,
  yesterdayStatus
)
```

Rules:

* Today greater than yesterday: `up`
* Today equal to yesterday: `equal`
* Today lower than yesterday: `down`
* Today missing: `unknown`
* Yesterday missing: `unknown`
* Today skipped: `unknown`
* Yesterday skipped: `unknown`

Do not skip backward several days to manufacture a “yesterday” comparison.

## 8.5 Attribute Direction

Compare the normalized state of each attribute today with its state yesterday.

```typescript
computeAttributeDirection({
  attributeCode,
  todayState,
  yesterdayState
})
```

Rules:

* Both days must contain a completed, non-skipped check-in.
* An attribute with no observation on a completed check-in inherits the pet’s baseline state.
* A missing check-in does not inherit baseline.
* A skipped check-in does not inherit baseline.
* Explicit option configuration must provide the attribute’s directional ordinal value.
* Do not infer clinical meaning from the ordinal comparison.

## 8.6 Direction Configuration

Observation options require two separate concepts:

```text
health_score_deduction
direction_ordinal
```

These values must not be treated as interchangeable.

Example:

| Vomiting option | Deduction | Direction ordinal |
| --------------- | --------: | ----------------: |
| No vomiting     |         0 |                 0 |
| Hairball only   |         0 |                 1 |
| Once            |         1 |                 2 |
| More than once  |         2 |                 3 |

This allows “Hairball only” to show movement without affecting the score.

# 13. Calculation Functions

## 13.1 Health Score Function

Create or replace the current scoring implementation with:

```typescript
type HealthScoreResult = {
  score: number;
  totalDeduction: number;
  deductionsByAttribute: Record<string, number>;
  reasonSummary: string;
};

function computeHealthScore(
  observations: ScorableObservation[]
): HealthScoreResult;
```

Required behavior:

1. Start at 10.
2. Include only configured Health Attributes.
3. Read `health_score_deduction`.
4. Group observations by attribute.
5. Cap each attribute’s deduction at 2.
6. Sum capped deductions.
7. Subtract deductions from 10.
8. Clamp the result to 0–10.
9. Return an integer.
10. Build the reason summary from attributes with deductions greater than zero.

## 13.2 Attribute State Resolver

Create:

```typescript
function resolveDailyAttributeState({
  attributeCode,
  checkIn,
  observations,
  baseline,
}): AttributeState;
```

Required behavior:

* Completed normal check-in plus no observation: baseline state.
* Completed changed check-in plus no observation for the attribute: baseline state.
* Matching observation: selected observation state.
* Missing check-in: unknown.
* Skipped check-in: unknown.
* Missing configured baseline: use the configured neutral/default state only when the Daily Check-In itself establishes “normal”; otherwise unknown.

## 13.3 Direction Function

Create:

```typescript
function computeAttributeDirection(
  todayState: AttributeState,
```

# 19. Business Rules

1. Baseline defines the pet’s individual normal.
2. Numerical scoring begins at 10 for every completed, scored day.
3. Only Health Attributes deduct points.
4. Wellbeing Attributes never change the score.
5. Weight never changes the score.
6. Each Health Attribute deducts no more than 2 points per date.
7. Missing data is unknown.
8. Skipped data is unknown.
9. No check-in is not equivalent to normal.
10. Direction represents movement, not medical judgment.
11. Historical observations must not be rewritten.
12. Legacy and V2 score semantics must remain distinguishable.
13. Only one Health Score snapshot may exist per pet/date.
14. Recalculation must be deterministic.
15. UI components must not calculate or persist scores directly.
16. Data access must use the existing entity/client layer.
17. Existing RLS and co-owner access must remain intact.

---

# 20. Validation Rules

## 20.1 Health Score

* Integer
* Minimum 0
* Maximum 10
* Required for normal check-ins
* Required for changed check-ins after successful calculation
* Null or absent for skipped check-ins

## 20.2 Deduction

* Integer
* Minimum 0
* Maximum 2
* Explicitly configured per observation option
* Capped at 2 per attribute/date

## 20.3 Direction Ordinal

* Integer or null
* Required for enum options used in directional comparison
* Must follow one consistent progression within an attribute
* Must not be shared across unrelated attributes as a clinical severity scale

## 20.4 Weight

* Must use the project’s existing valid numeric range and unit conversion rules
* Must not create a Health Score deduction
* Must not use a missing value as zero

## 20.5 Attribute Codes

Only configured Health Attribute codes can affect scoring.

Unknown codes must be ignored and logged.

# 24. Edge Cases

2. Duplicate observations exist for one Health Attribute.
3. A mild and significant observation exist for the same attribute.
4. A Wellbeing option accidentally has a deduction value.
5. Weight exists without a check-in.
6. Check-in exists without yesterday’s check-in.
7. Yesterday was skipped.
8. Today was changed to skipped after a score was saved.
9. Today was skipped and later changed to normal.
10. A score calculation is retried.

12. A baseline is incomplete.
13. Observation option configuration is missing.
14. An unknown observation type is returned.
15. The Pets Wellbeing request fails while pet identity succeeds.
16. Weight values use different display units but normalized stored values.
17. Two co-owners edit the same check-in.
18. A memorial pet has historical score data.
19. The user changes timezones near midnight.
20. A check-in date differs from the UTC creation date.
