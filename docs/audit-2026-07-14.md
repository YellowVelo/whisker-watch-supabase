## Step 4 — Orphan Check

- **`docs/launch-punch-list.md` P3 section** describes severity-scoring mechanics (`observation_options.severity_score`, equal-weight multi-select deductions, migration 0022 as "the" scoring model) that were **replaced eight days after being written** by the Vibe/symptom-count model (migrations 0026–0027). The doc's own text ("Historical Health Score backfill — done") now describes a data model that migration 0027 further migrated away from.
- **README.md's account-deletion note** is explicitly called out as stale by the punch list itself (see conflict row above) — a rare case of one doc correctly flagging another doc's staleness, but neither has been fixed.
- **`docs/launch-punch-list.md`'s manifest.json item** references a file that no longer needs to exist as a static asset — it's now generated at build time. The item is unchecked as if still outstanding.
- No doc references features/fields that were fully deleted from the schema (the old scoring columns are deliberately kept in place per migration comments), so there's no "points to a dropped column" case — the staleness is entirely about which *model* is currently authoritative, not about broken references to nonexistent data.

## Step 5 — Highest-Risk Gaps Before Alpha

1. **The actual governing spec for the Vibe/Trends model isn't in the repo.** Migrations 0026/0027 quote it constantly ("Feature Spec v5," "Core Model III," "Migration Plan," "Edge Cases") as ground truth, but nobody opening this repo cold can find it. If that external doc is lost or inaccessible, the *reasoning* behind the current data model (why Vibe and symptom count are never derived from each other, why certain fields stay null) disappears — only the code remains.
2. **The scoring model has churned 4 times in 8 days (07-05 → 07-09 → 07-11 → 07-13) and the only in-repo doc describing it (`docs/launch-punch-list.md` P3) is already one generation behind.** Anyone using that doc to understand "how health scoring works" right now will describe a model that no longer runs.
3. **Two docs actively contradict each other on account deletion** — README says accounts aren't really deleted; the punch list says that's now false and flags README as stale. Shipping to Alpha with a live contradiction about a data-deletion / privacy-relevant behavior is a bad look if a user or reviewer notices, and it's a five-minute fix once someone decides which is true.
4. **Cloudflare deploy config and the manual-deploy gate are undocumented and partly outside the repo entirely.** `wrangler.jsonc` exists with zero README coverage, and the 07-12 "verify manual-deploy gate" commit implies a gate configured in the Cloudflare dashboard that has no representation in-repo at all. If the person who set that up is unavailable, nobody else knows deploys are gated or how.
5. **The transactional email system (Resend) and its two security fixes are entirely undocumented**, despite touching auth-adjacent surfaces (password reset, invite flows) and having already needed a JWT-auth fix and a CTA-domain-allowlist fix — exactly the kind of subsystem where undocumented assumptions cause the next security fix to regress the first one.
