Status

Overview and Trends are implemented. Patterns and Compare remain placeholder screens.

Version History

v2 — Original spec. Only Overview was implemented; Health Score examples still showed the legacy 0-100 scale; Trends was listed as a tab but never given defined content; trend calculations were specified to compare against a per-pet baseline.

v3 (this version) — Reconciles the spec with what was actually built, and defines Trends for the first time. Health Score is 0-10 throughout (V2 scale, shipped separately). Trends is now a real, fully-specified screen, not a placeholder or a duplicate of Overview. The per-pet-baseline comparison rule is dropped (see Business Rules). Reflects the move to multi-select, equal-weight symptom logging across all 9 Health/Wellbeing attributes.

Purpose

Provide a dedicated screen for visualizing changes in a pet's health over time.

The Trends screen presents historical health metrics using charts, summaries, and AI-generated insights.

This screen is read-only.

No health data is entered from this screen.

6. Trends Tab — Attribute Group Charts

Unlike Overview (which covers only Health Score, Appetite, Water Intake, Energy, Weight, and Insight Summary), the Trends tab makes every Health and Wellbeing attribute available as its own chart.

Attributes are organized into two groups, switchable via an in-page toggle:

Health group (6): Appetite, Water Intake, Bathroom, Stool, Vomiting, Weight
Wellness group (4): Energy, Mobility, Breathing, Skin/Itching

Each attribute in the active group renders its own chart card, using the same chart component and legend as Overview's observation cards (Section 5) — no separate chart style for Trends.

The group toggle defaults to whichever group the entry point implies (see Deep-Linking below); a user who lands on Trends via the tab itself (not a deep link) sees the Health group by default and can switch to Wellness.

Users can scroll through every chart in the active group.

Deep-Linking

A tap on a specific attribute chip (Home's Health Attribute chips, or Pets'/Pet Profile's Wellbeing chips) opens Trends already on the correct group, scrolled to and briefly highlighting that attribute's chart — the user should never have to hunt for the metric they tapped.

Navigation One (Health):

Home (taps a Health Attribute chip, e.g. Appetite)

↓

Trends, Health group, scrolled to Appetite

Back returns to Home. Selected pet remains unchanged.

Navigation Two (Wellness):

Pets or Pet Profile (taps a Wellbeing chip, e.g. Mobility)

↓

Trends, Wellness group, scrolled to Mobility

Back returns to Pets (or Pet Profile, whichever the tap originated from). Selected pet remains unchanged.

Empty States

No Health Data

Display: "No health trends available yet."

Primary action: Complete today's Daily Check-In.

No Weight

Display: "No weight history available."

No Insight

Display: "Complete more check-ins to unlock AI insights."

Loading States

While loading: display skeleton cards. Charts display placeholders. Cards load independently.

Changing date range or attribute group displays loading only within chart content — the screen itself never reloads.

Error States

If metric retrieval fails: display inline message, "Unable to load trend." Retry automatically when the user changes the date range. Other cards continue functioning.

If AI summary fails: hide summary, display "Insights unavailable." Other cards continue functioning.

Business Rules

Severity is flat and global. Every symptom logged within a category carries equal weight, regardless of which specific answer was selected — an owner is reporting an observation, not providing a clinically graded assessment. There is no per-option severity ranking (e.g. "Diarrhea" and "Softer than usual" are not weighted differently).

Trend calculations do NOT compare against a per-pet baseline. (Superseded from v2 — the original "always compare against the pet's baseline where available" rule was never actually implemented and conflicts with the flat/global severity model above. `pet_baselines` remains populated as an informational reference only, never as a scoring or trend-comparison multiplier. Revisit only if per-pet baselines become a real scoring input in a future feature.)

Every multi-select category (the 5 Health Attributes + the 4 Wellbeing Attributes) allows more than one symptom to be logged per day. A day's chart bar and legend reflect the count of distinct symptoms logged (0, 1, 2+), not a graded direction.

Every completed (non-skipped) check-in writes an explicit record for each multi-select attribute, every day — a confirmed-normal baseline record when nothing was selected, one record per distinct symptom otherwise. Absence of a record is never interpreted as an assumption.

Raw observation data is never modified after the fact. Computed/derived values (e.g. a stored Health Score) may be recomputed via an explicit, versioned migration when the scoring rule itself changes — as was done once, deliberately, when the equal-weight model shipped — but never silently or automatically as part of routine operation.

Unknown days remain gaps.

Skipped days are visually distinguishable from missing data.

Trend summaries never diagnose.

Charts display only owner-accessible data.

Future dates are never shown.

Weight changes use recorded measurements only.

AI summaries describe observations. They never recommend diagnosis or treatment.

Validation Rules

Time range must be one of: 24H, 7D, 30D, 90D, 1Y.

Metric values must remain within valid ranges.

Health Score: N/10.

Weight: positive decimal.

A multi-select attribute's daily deduction (Health Attributes only) is capped at 2, computed as min(2, count of distinct non-baseline symptoms logged that day).

A day's total Health Score deduction is the sum of the 5 Health Attributes' individual capped deductions; the resulting score ranges 0–10.

Behavioral observations must map to supported observation values. Unknown values render as "No Data."

Data Requirements

Retrieve:

Pet
Health Scores (wellness_scores.health_score, V2)

Observations, including symptom counts per attribute per day (derived by excluding baseline values, not read from a graded severity column)

Use the existing entity layer. Never query Supabase directly from UI components.

Support owner-scoped RLS.

Charts must respect the currently selected time window.

Health/Wellness group membership for the Trends tab is derived from the single canonical attribute list (Health Score Attributes / Wellbeing Attributes) already defined in the data layer — never re-listed ad hoc in the UI.

Acceptance Criteria

Users can:

✓ Open Trends from Pets, Pet Profile, or Home.

✓ See the Trends tab as a real, distinct screen — not a copy of Overview.

✓ Switch between the Health and Wellness attribute groups on the Trends tab.

✓ Switch between all supported time ranges.

✓ View historical Health Score.

✓ View Appetite, Water Intake, Bathroom, Stool, Vomiting, and Weight history (Health group).

✓ View Energy, Mobility, Breathing, and Skin/Itching history (Wellness group).

✓ Log 2+ symptoms for one category on the same day and see the chart reflect a "2+ Symptoms" state, not a graded direction.

✓ Tap a Health Attribute chip on Home and land on Trends, Health group, scrolled to that attribute.

✓ Tap a Wellbeing chip on Pets/Pet Profile and land on Trends, Wellness group, scrolled to that attribute.

✓ Navigate back and land on the actual screen they came from, not a hardcoded intermediate page.

✓ Read Insight Summary.

✓ View charts that update without reloading the page.

✓ See appropriate empty states.

✓ Continue using the screen if one metric fails to load.

✓ View only data belonging to pets they are authorized to access.

Edge Cases

First Daily Check-In completed today.
Only one historical data point exists.
All historical values are identical.
Missing days within the selected range.
Skipped check-ins within the selected range.
A category with 2+ distinct symptoms logged the same day.
Pet has no weight history.
Pet has no observations.
AI summary unavailable.
Date range returns no records.
Rapid switching between date ranges or attribute groups.
Extremely long histories (>1 year).
Memorial pets remain viewable in read-only mode.
A deep link's metric parameter names an attribute not in the linked group (stale/malformed link) — screen loads normally without scrolling to a nonexistent card, no crash.
Historical days recomputed under a rule change (e.g. the equal-weight migration) may show a different score than what was originally displayed live that day — this is expected and not a data integrity issue.

Implementation Notes for Claude Code

Preserve the existing visual style, spacing, typography, and card patterns defined in the Design System.

Implement Trends using the same reusable chart infrastructure Overview already uses (TrendChart / ObservationCard / ObservationLegend, or equivalent) — configuration changes only, never a second chart implementation.

Route all data access through the existing entity layer and data services. Do not access Supabase directly from UI components.

Build each metric card as an independently loading component so failures in one section do not block rendering of the rest of the page.

Health/Wellness group membership must be derived from the app's single canonical attribute lists, not re-listed inline in the Trends page.

Deep-link state (active tab, active group, and the highlighted metric) lives in the URL query string, not local component state, so back/forward navigation and shared links behave correctly.

Treat the Trends page as a read-only visualization layer. No editing, logging, or baseline management should occur from this screen.

Keep the Overview, Patterns, and Compare tab architecture reusable so future implementations can be added without restructuring the page.

Ensure charts correctly distinguish between normal observations, skipped days, and missing data, and correctly reflect symptom counts (not a graded severity) for every multi-select attribute.

Maintain smooth chart updates when switching time ranges or attribute groups without reloading the entire page.
